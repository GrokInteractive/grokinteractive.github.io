Make your emails look right everywhere

by Anton Domratchev    Oct 12, 2014    Software development Email, Outlook, Cross mail client, Compatibility

When you are faced with creating an email blast there is a number of things to keep in mind. Why are emails look right in one email client but not the other? How can you test that the changes you make don't break something else? The purpose of this blog post is to help you make your emails look right across all email clients and also give you some great tools to help with the testing. I will discuss some Do's and Don'ts of HTML emails, HTML email boilerplate, and a great platform to test your emails and share some tips and tricks to make your email template troubles go away.

Do's and Don'ts

Though, email clients support HTML and CSS not every style and HTML element is supported. Luckily guys at Campaign Monitor did all the hard work of testing and compiled a The Ultimate Guide to CSS. This guide gives us a good range of styles we can use. Now that we know which styles are available for us to use we can now set up a baseline for our email templates. When I was faced with optimizing emails specifically for Microsoft Outlook, specifically 2007, 2010, and 2013 I noticed that HTML is not being rendered the same as the browser. In fact, earlier versions of Outlook from 2000 to 2003 render HTML emails better than the newer versions. This is due to the change Microsoft implemented with Office 2007. The HTML rendering engine for Office (previously Internet Explorer) was changed to Microsoft Office rendering engine. Which relies more on the <mso-tags> than the actual HTML. So keep that in mind when optimizing for Outlook. Note that some email clients strip styles and external resources.

Email Boilerplate

You may be aware of the HTML boilerplate that is used to baseline HTML and styles for a page. Good news is there is a boilerplate for emails also. There are several email boilerplates available on-line here is the one I prefer to use because it uses MailChimp CSS Guide and also includes mobile targeting. There are several sections of stylesheet reset that you can use. Starting with some baselines from MailChimp:

    /* Based on The MailChimp Reset INLINE: Yes. */

    /* Client-specific Styles */
    /* Force Outlook to provide a "view in browser" menu link. */
    #outlook a {
        padding:0;
    }

    body {
        width:100% !important;
        -webkit-text-size-adjust:100%;
        -ms-text-size-adjust:100%;
        margin:0;
        padding:0;
    }

    /* Prevent Webkit and Windows Mobile platforms from changing default font sizes.*/
    /* Force Hotmail to display emails at full width */
    .ExternalClass {
        width:100%;
    }
    .ExternalClass,
    .ExternalClass p,
    .ExternalClass span,
    .ExternalClass font,
    .ExternalClass td,
    .ExternalClass div {
        line-height: 100%;
    }

    /* Forces Hotmail to display normal line spacing.  More on that: http://www.emailonacid.com/forum/viewthread/43/ */
    #backgroundTable {
        margin:0;
        padding:0;
        width:100% !important;
        line-height: 100% !important;
    }

    /* End reset */

Now all we have to do is build our email template and test it in every email client that's available. Sounds like a very time consuming task, fear not and read on.

Testing your email cross client compatibility

When it comes to cross mail client compatibility it is most often overlooked in favor of optimizing emails for a popularly used email client such as Gmail or Thunderbird. There are literally dozens of different email clients and platforms that need to be tested to make sure it look right on every email client. The headache involved with making sure emails look right across all of these email clients is what discourages most developers from trying to optimize their emails and simply stick with one that works. Here is where Litmus comes in. Litmust, as you may have guessed, is a testing platform for email clients. "Litmus lets you preview your emails across 30+ real email clients and devices in minutes." Litmus works with either the HTML body of your email or an actual email you send to the Litmus testing platform, and in minutes you are viewing your email in nearly every popular email client rendered within a virtual machine. The neat part about Litmus is you can isolate which client is having issues displaying your email correctly and go back to fix those issues. With every new version of email you create, each client will re-render the email and you can see how you changes affected other clients.

Tips and tricks

Armed with the boilerplate for your HTML email baseline and Litmus for your cross client compatibility testing, you are well on your way to create your awesome email template. Along with those tools I would like to give you are couple of tips and tricks I learned while creating my email templates.

  1. When using Litmus to test your email template, specifically, the Outlook clients Litmus allows you to preview the HTML as it is rendered within the Outlook client. Using that preview you can also make edits to the rendered HTML and preview them live. This great feature saved me a lot of time testing and debugging issues with Outlook.

  2. Avoid using certain CSS properties that you regularly use in HTML pages. Some of the things include rounded corners of elements, background images, box-shadows. The previously mentioned properties are completely not supported in any version of Outlook so if you care that your email will look the same in all email clients, avoid using them. There are several other CSS properties that are not supported in Outlook, which you can find in The Ultimate Guide to CSS. Keep this guide close to you when choosing styling for your email elements.

  3. While optimizing my emails for Android, specifically for Gmail App on Android, I ran into the issue of certain elements overflowing and being cutoff by the Android view port. This is a known issue with Gmail App for Android. So if you are finding your emails being cut off and your width parameters being completely ignored, I have a workaround for you. You can force Gmail App for Android to display your element's full width by inserting a blank image spacer (such as this spacer.gif) into the container element that is being cutoff. Using this small image you can set the width and height properties to match that of your container block that you want to force, and Gmail App for Android will render the container according to the size of the image within it.
