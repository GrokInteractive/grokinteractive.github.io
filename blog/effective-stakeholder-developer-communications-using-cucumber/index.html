Effective Stakeholder / Developer Communications using Cucumber

by Jason Ellis    Sep 08, 2015    Software development, Business development Developers

There are not too many things more critical to a successful project than effective bi-directional stakeholder / developer communications. In fact, I can think of very few things that will sink a project faster.

I also can’t think of a single area during the life cycle of a given project that not only lasts the entire length of the project but that is also more rife with struggle, confusion, frustration and just generally fraught with issues.

The focus of these communication gaps are often centered around feature requests, enhancements, expectations and functionality that can often get defined and redefined dozens of times over the course of a short period, usually resulting on no one being on the same page at the same time. Or worse, active development trying to hit a moving target. Developers and stakeholders alike get frustrated to the point of breaking.

If this sounds like the voice of experience it’s because it is. I’m sure we’ve all been there on one side or the other, and it’s very easy to fall into the trap of asking yourselves, “Who’s right?”

What I have learned is that it honestly doesn’t matter! We need to stop trying to figure out who is right and who is wrong and start working more towards shoring up the communication shortcomings or in the end no one will be happy.

Core to the issue however is that developers and stakeholders approach this communication process differently; be the individual a CEO/CTO, manager/PM, coder, technical writer or your grandmother, we all come at a problem from a different angle which likely is what led to the problem in the first place.

At Grok we recognized fairly early that we needed a communication method that would allow everyone to firmly define the needs and requirements of an application. And do it in such a way as to unify the team as a whole instead of divide us.

This new communication method had to not only be easily digestible by everyone involved, but it also had to be directly actionable by the developers.

Enter Cucumber.

The Cucumber project was started in April 2008 by Aslak Hellesøy. The core concept of which is to create small text files that describe features, scenarios and user stories in such a way as to be very simple and easily digestible by stakeholders.

As an example, an application that has a user login needs to be able to handle a user entering the wrong password. This is something that we all take for granted and just assume it will give a standard flash message at top proclaiming the error, and move on.

But unless you actually check with the client, you won’t know if that’s what they want. They might have a completely different idea of how it should work.

So lets write a simple sentence of how we think it should work and convert that to the DSL; "In order to get access to a protected section of the site a user should be able to sign in and if they exist as an approved user but they try to sign in with an invalid password, show an invalid login message."

The feature of that should look something like this;

Feature: Sign in
  In order to get access to protected sections of the site
  As a user
  I should be able to sign in

  Scenario: User enters wrong password
    Given I exist as an approved user
    And I am not logged in
    When I sign in with a wrong password
    Then I see an invalid login message

I can pass that block of text (and dozens more like it) to a client for each feature and make sure that everyone is on board with the process and how this particular feature should work.

This simple act of putting plain English to a feature brings everyone on the same page by describing an application behavior in a very simple business-readable domain-specific plain text.

This was obviously a very simple example but you can get as complex as any application you can dream up. It gives each feature a plain text description of how a they should behave and affords the stakeholders the opportunity to modify and adjust the features descriptive language in the same (or similar) plain text until the feature works as they expect and now how we assumed.

The best part is that once both sides have gone back and forth on this particular feature and it is blessed by all the powers that be, the feature file is directly actionable by the developers! A simple step definition file can be written in Ruby, Java, .NET, etc... that interprets the feature plain text into a business driven development test that the developers can code to.

Win!

In one simple additional step to the software development process, you get a fully defined feature in written form that can serve not only as signed and blessed documentation of said feature, but as an active test that makes sure the feature works and gives the programmers something to write towards.

The end results of which are that everyone is now approaching the feature from the same angle, talking the same language, and on the same page. Communication has never been more effective.

TL;DR

Effective stakeholder / developer communications is vital to a successful relationship. Using a tool like Cucumber can actually help greatly by getting everyone on the same page.
